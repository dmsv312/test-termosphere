# AI_USAGE.md — как использовали AI

Проект вёлся в формате AI-first: разработчик ставил задачи, принимал решения и проверял результат; AI генерировал код по спецификации. Этот файл дистиллирован из живого журнала [docs/worklog.md](docs/worklog.md) и отвечает на пять вопросов: какими инструментами пользовались, какие задачи отдавали AI, что проверяли вручную, где AI ошибался или мог ошибиться, какие решения принимали сами.

Принцип честности журнала: записи «ПРОВЕРКА» отражают только то, что реально запускалось и сверялось.

## 1. Инструменты

- **Claude Code** — основной инструмент: генерация кода по шагам, рефакторинг, прогон тестов, независимые ревью-проходы по диффу, Playwright-снимки живого контура.
- **ChatGPT Codex (OpenAI Codex)** — вспомогательно.

Других ассистентов не использовали.

## 2. Какие задачи отдавали AI

Работа шла по фазам (см. «План внедрения» в [AGENTS.md](AGENTS.md)); на каждом шаге AI получал узкую задачу с приёмкой по факту:

- **Каркас** — docker-compose (Postgres), Makefile, скелет FastAPI (config/session/main), `.env(.example)`.
- **raw-слой** — 13 моделей `raw_*` (все колонки TEXT, суррогатный PK, без ограничений) + идемпотентный загрузчик CSV → raw + миграция Alembic.
- **core-слой** — модели SQLAlchemy с настоящими типами, PK/FK/CHECK; наши добавления `sources` (справочник каналов) и `data_quality_issues` (журнал проблем); сид справочника в миграции.
- **transform** — чистые функции нормализации (`clean.py`: канал/дата/телефон/сумма/ИНН/bool) + обобщённый проход raw → core (`transform.py`: дедуп, проверка связей, три реакции fix/quarantine/flag с логом) + юнит-тесты чистилок.
- **Живой контур** — FastAPI-роуты поверх core (витрина по белому списку + отчёт «проблемы данных»), React-дашборд (Vite + Recharts), docker-сборка api + web(nginx), basic-auth, публикация за своим Cloudflare-туннелем.
- **Семь отчётов** — вертикальными срезами SQL → API-эндпоинт → страница; канонический SQL всех отчётов в одном месте (`reports/queries.py`), он же исполняется API и выгружается в `db/reports.sql`.

## 3. Что проверяли вручную (реальные проверки)

Приёмка «на словах» не принималась — каждый шаг сверялся:

- **Полнота загрузки:** независимый CSV-парсер по каждой из 13 таблиц против БД — совпало построчно, итого **80 = 80** строк; спот-чек, что «грязь» сохранена в raw как есть (кривые даты, пустая валюта, дубль D1008, отрицательная сумма, сирота D9999).
- **Эталонные расчёты (Приложение B)** прогнаны вручную ДО реализации и затем сверены с SQL: маржа по позициям (D1001=39100 … D1005=48500), дебиторка (остаток 436 000; D1010=169 000 с корректировкой −5000, D1005=−8 000 переплата), воронка (Σ 1 040 000, WON=2/361 000), задержка производства (PO003=6 дн), источники (yandex_direct ROMI +2778 %, avito −100 %), дата среза = 2026-06-15.
- **Ограничения БД работают:** FK отклонил позицию на несуществующую сделку; CHECK отклонил недопустимый статус; обратимость миграций (`downgrade -1` → `upgrade head`).
- **Живой контур глазами:** Playwright-снимки всех 7 страниц отчётов + просмотра core — 0 JS-ошибок; fullPage-скриншоты просмотрены визуально (столбцы графиков, подсветка переплаты/неактивного менеджера/флаг-строк, интерактивный параметр N). Сквозная цепочка через nginx и публично по HTTPS с basic-auth.
- **Тесты чистилок:** `pytest` (23 зелёных) — регистр/алфавит каналов, ISO vs кривые даты, телефон 8→+7, Decimal, ведущие нули ИНН.

## 4. Где AI ошибался или мог ошибиться (и как поймали)

Показательные случаи — почему проверка обязательна:

- **FSM ложно пометил легитимную сделку.** Строгая карта «в WON только через SHIPPED» отметила переход D1003 `PROPOSAL → WON` как нарушение, хотя сделка выиграна законно. Исправлено обобщённо: терминальные WON/LOST достижимы из любой активной стадии; нарушение — только форвардные прыжки через стадии (реально остался лишь D1001).
- **Значения вне CHECK-домена пробрасывались сырыми** (типы оплат/активностей/статусы производства). На этой выгрузке прошло, но на других данных уронило бы весь проход по CHECK. Добавлены домены-константы: значение вне домена → NULL + лог `invalid_*`. Проверено впрыском мусорных значений в raw.
- **Порядок вставки по FK.** У core-моделей нет `relationship()`, один commit не гарантировал порядок между таблицами → `ForeignKeyViolation`. Исправлено: загрузка в FK-безопасном порядке с `flush()` по уровням.
- **Даты-«сутки» уезжали на день.** Чисто-датные поля лежали в `timestamptz` как полночь +05 и при `::date` в UTC откатывались назад, ломая дату среза. Переведены в тип `Date`.
- **Недетерминированный вывод.** Дедуп сделок мог занести победителя в карантин при непарсящейся дате; порядок агрегатов в отчёте о качестве был нестабилен. Переписано на детерминированную группировку/сортировку.
- **UX-регресс в поле N.** Атрибут `max` ограничивал только стрелки; набранное вручную значение > 3650 уходило на API и ловило 422. Добавлен зажим в `onChange`.

Все находки — из ручных сверок и отдельных ревью-проходов по диффу, не из «зелёных тестов».

## 5. Какие решения принимали сами (человек)

AI не выбирал бизнес-логику — её фиксировал разработчик (обоснование — в [ASSUMPTIONS.md](ASSUMPTIONS.md)):

- **Сумма сделки** = `COALESCE(Σ позиций, expected_amount)`; маржа — только по позициям. В базе храним `expected_amount` (факт из карточки), гибрид считаем в отчётах — не запекаем производную величину.
- **Дебиторка:** учитываем обязующие стадии ИЛИ сделки со сделанными оплатами; `correction` уменьшает долг (остаток = сумма − оплачено + Σ correction); `pending` — отдельной колонкой; LOST исключаем.
- **Дата среза** = максимум по фактическим событиям (без будущих плановых дат) = 2026-06-15; «касание» для отчёта «без активности» = только записи активностей.
- **ROMI** при нулевых затратах = «нет затрат» (не делим на ноль). `discount` — абсолютная сумма в рублях. WON без даты закрытия — флаг (дату не выдумываем).
- **Архитектура и процесс:** слои raw → core + `sources` + `data_quality_issues`; три реакции fix/quarantine/flag; FastAPI + SQLAlchemy + Alembic без pandas; React/Vite + Recharts; Postgres в Docker; отдельный репозиторий; ранний живой контур и деплой за своим Cloudflare-туннелем (соседние проекты не трогаем); публичный доступ закрыт basic-auth.
- **Порядок фаз** пересмотрен под живую верификацию: контур и деплой поднимаем рано, 7 отчётов делаем вертикальными срезами со сверкой по ходу.
