# dashboard_plan.md — API и дашборд

План API/дашборда из чек-листа. Контур реализован и опубликован (живой стенд — за basic-auth, см. [README](README.md)), поэтому ниже — описание того, что фактически построено, а не только замысел.

## Принципы

- **Read-only.** Дашборд ничего не меняет в БД — только читает core и отчёты. Никаких мутаций через API.
- **Один origin, без CORS.** Фронт всегда зовёт относительный `/api/*`: в dev его проксирует Vite, в проде — nginx на контейнер API. CORS-возни нет.
- **Источник чисел — SQL.** Отчёты считаются каноническим SQL (`backend/app/reports/queries.py`); тот же SQL выгружается в сдаваемый `db/reports.sql`. API и файл не расходятся по определению.
- **Деньги и ИНН — строками.** Decimal сериализуется в строку (точность денег и ведущие нули ИНН не теряются), даты — ISO, счётчики/разницы дат — числами.
- **Витрина core — по белому списку.** Можно запросить только известные модели; произвольную таблицу — нельзя (неизвестная → 404).

## API (FastAPI, Swagger на `/docs`)

| Метод, путь | Отдаёт |
|---|---|
| `GET /health` | статус приложения и доступность БД |
| `GET /api/core/tables` | список core-таблиц: имя, подпись, число строк |
| `GET /api/core/{table}` | строки таблицы по белому списку (колонки + данные, флаг качества на строке) |
| `GET /api/data-quality/summary` | агрегат проблем: всего, по реакции (fixed/quarantined/flagged), по типам × реакция |
| `GET /api/data-quality/issues` | детальный список всех проблем качества |
| `GET /api/reports/funnel` | отчёт 1 — воронка: количество и сумма сделок по стадиям |
| `GET /api/reports/managers` | отчёт 2 — продажи и маржа по менеджерам (по WON; SHIPPED — отдельной пометкой) |
| `GET /api/reports/receivables` | отчёт 3 — дебиторка: сумма, оплачено, в ожидании, корректировка, остаток |
| `GET /api/reports/production-delays` | отчёт 4 — заказы с задержкой производства > 5 дней |
| `GET /api/reports/stale-deals?n_days=N` | отчёт 5 — живые сделки без активности больше N дней (N — параметр) |
| `GET /api/reports/sources` | отчёт 6 — источники: выручка/маржа и окупаемость (ROMI) |

Отчёт 7 «Проблемы данных» обслуживают ручки `data-quality/*`.

## Дашборд (React + Vite + Recharts)

Два верхних раздела:

**1. Отчёты** — под-навигация, страница на каждый отчёт:

| Страница | Что показывает |
|---|---|
| Воронка | столбчатый график сумм по стадиям (WON зелёным, LOST красным) + таблица со «Всего» |
| Менеджеры | продажи и маржа по менеджерам; неактивный менеджер и «отгружено, не закрыто» помечены |
| Дебиторка | таблица остатков; переплата (отрицательный остаток) подсвечена |
| Производство | заказы с задержкой > 5 дней от даты среза |
| Без активности | таблица + интерактивное поле **N, дней** (зажато в диапазон) |
| Источники | график выручки/затрат по каналам + таблица с ROMI («нет затрат», где затрат нет) |
| Проблемы данных | карточки (всего / fixed / quarantined / flagged), график по типам × реакция, таблица всех проблем |

**2. Просмотр core** — сайдбар со всеми 15 core-таблицами (и счётчиками) + грид выбранной таблицы; строки с бизнес-сигналом (флаг качества) подсвечены.

Графики — с выключенной анимацией (`isAnimationActive=false`): детерминированный рендер, в том числе под headless-снимки.

## Деплой

docker-compose (db + api + web). API — uvicorn в контейнере; web — nginx, отдаёт SPA и проксирует `/api` на контейнер API (SPA-фоллбэк на `index.html`). Весь публичный доступ закрыт basic-auth на уровне nginx (включая `/api`). Публикация — за собственным Cloudflare-туннелем; соседние проекты не затрагиваются. Порядок первого запуска на чистом стенде — в [README](README.md).

## Metabase (BI-бонус, Шаг 7)

Поверх той же БД поднят **Metabase** (v0.62.3.5) — второй, «почти без кода», интерфейс к тем же данным. Реализован как отдельный docker-профиль `bi` (по умолчанию не поднимается — `make up-full` его не трогает).

**Изоляция и ресурсы:**
- Метаданные Metabase — в отдельном контейнере `metabase_db` (том `mbdata`), **не** в CRM-томе `pgdata`. Боевая БД `termosphere` подключена как **источник данных** (read-only-использование).
- JVM heap ограничен (`-Xmx1g`) — чтобы не давить соседей на общем сервере.

**Доступ:** через nginx-прокси `metabase_proxy` (заголовки/websockets/таймауты), опубликован за Cloudflare-туннелем. Авторизация — собственным логином Metabase (email+пароль); отдельного basic-auth на BI-хостнейме нет, то есть один запрос авторизации. Сам Metabase напрямую наружу не торчит (хостовый порт — только `127.0.0.1`, для провижининга/админа).

**Провижининг автоматический** (`scripts/metabase-provision.mjs`, идемпотентно через REST API Metabase): создаёт администратора, подключает источник `termosphere`, синкает схему и создаёт 7 вопросов из **того же SQL, что и отчёты** (`db/reports.sql`) + собирает дашборд «ТермоСфера — обзор». Числа сверены с эталонами: воронка Σ 1 040 000, дебиторка остаток 436 000, источники revenue 361 000.

**Запуск:**
```bash
make bi-up          # поднять metabase_db + metabase + прокси (профиль bi)
make bi-provision   # авто-настройка (admin + источник + вопросы + дашборд)
```

Дальше в самом Metabase дашборды и вопросы собираются мышью без кода — это и есть смысл бонуса поверх готовой нормализованной модели.
