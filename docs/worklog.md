## 2026-07-01 · Проектирование (до кода)

**РЕШЕНИЕ** — Backend: FastAPI + SQLAlchemy + Alembic (без pandas).
  Почему: лёгкий async-фреймворк с авто-документацией API (Swagger); SQLAlchemy-модели = целевая схема, Alembic — миграции; для маленькой ETL-задачи pandas избыточен, CSV читаем обычным Python.
  Отклонено: Django (batteries-included, но здесь не нужны его ORM/admin/шаблоны в комплекте).

**РЕШЕНИЕ** — Слои `raw (TEXT, без ограничений) → core (типы, PK/FK/CHECK)` + справочник `sources` + таблица `data_quality_issues`.
  Почему: выгрузка намеренно содержит грязные данные; нельзя ронять импорт кривой датой/суммой и нельзя молча терять строки — каждую проблему логируем.

**РЕШЕНИЕ** — Проверка связей и решения (fix/quarantine/flag) — в Python-transform до вставки; ограничения БД в core — страховка целостности.

**РЕШЕНИЕ** — Зафиксированы бизнес-решения по отчётам (сумма сделки = гибрид; дебиторка = только `paid`; «без активности» N=14; скидка = рубли; WON без даты — флаг). Обоснование — в `ASSUMPTIONS.md`.

**ПРОБЛЕМА** — Предварительный разбор выгрузки: найдено 28 дефектов качества (полный каталог — `DATA_QUALITY.md`). В т.ч. два неочевидных, выявленных сверкой расчётов: переплата по сделке D1005 (оплачено 100000 при сумме 92000) и производственный заказ PO003, созданный на 4 дня раньше своей сделки.

**ПРОВЕРКА** — Эталонные расчёты (выручка/маржа по позициям, дебиторка paid-only, задержки производства) прогнаны вручную до реализации — будут использованы для сверки будущих SQL-отчётов.
