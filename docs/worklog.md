## 2026-07-01 · Проектирование (до кода)

**РЕШЕНИЕ** — Backend: FastAPI + SQLAlchemy + Alembic (без pandas).
  Почему: лёгкий async-фреймворк с авто-документацией API (Swagger); SQLAlchemy-модели = целевая схема, Alembic — миграции; для маленькой ETL-задачи pandas избыточен, CSV читаем обычным Python.
  Отклонено: Django (batteries-included, но здесь не нужны его ORM/admin/шаблоны в комплекте).

**РЕШЕНИЕ** — Слои `raw (TEXT, без ограничений) → core (типы, PK/FK/CHECK)` + справочник `sources` + таблица `data_quality_issues`.
  Почему: выгрузка намеренно содержит грязные данные; нельзя ронять импорт кривой датой/суммой и нельзя молча терять строки — каждую проблему логируем.

**РЕШЕНИЕ** — Проверка связей и решения (fix/quarantine/flag) — в Python-transform до вставки; ограничения БД в core — страховка целостности.

**РЕШЕНИЕ** — Зафиксированы бизнес-решения по отчётам (сумма сделки = гибрид; дебиторка = только `paid`; «без активности» N=14; скидка = рубли; WON без даты — флаг). Обоснование — в `ASSUMPTIONS.md`.

**ПРОБЛЕМА** — Предварительный разбор выгрузки: найдено 28 дефектов качества (полный каталог — `DATA_QUALITY.md`). В т.ч. два неочевидных, выявленных сверкой расчётов: переплата по сделке D1005 (оплачено 100000 при сумме 92000) и производственный заказ PO003, созданный на 4 дня раньше своей сделки.

**ПРОВЕРКА** — Эталонные расчёты (выручка/маржа по позициям, дебиторка paid-only, задержки производства) прогнаны вручную до реализации — будут использованы для сверки будущих SQL-отчётов.

## 2026-07-01 · Уточнения по итогам обсуждения с владельцем

**РЕШЕНИЕ** — Дебиторка: платёж типа `correction` уменьшает долг, а не «оплачено». Остаток = `сумма − оплачено + Σ correction` (correction со знаком). D1010: единственная запись `correction −5000` при сумме 174000, обычных оплат нет → остаток 169000. В отчёте — отдельная колонка «корректировка».
  Отклонено: прежняя трактовка «минус уменьшает оплачено» (давала остаток 179000 — долг больше суммы сделки, бессмыслица). Правит `ASSUMPTIONS §2`.

**РЕШЕНИЕ** — Дата среза («сегодня») = **2026-06-15** = максимум по фактическим событиям (`updated_at/payment_date/actual_date/completed_at/changed_at`), без будущих плановых дат (`custom_deadline`, `planned_finish`). Иначе PO001/D1001 (план 12.06, факта нет) ложно попал бы в «задержку производства». Одна дата среза на отчёты 4 и 5.

**РЕШЕНИЕ** — Маржа/продажи по менеджерам — по WON; SHIPPED (D1005 — отгружена и оплачена, но не закрыта) показываем отдельной пометкой «отгружено, не закрыто»: не теряем фактическую продажу и не смешиваем с закрытыми.

**РЕШЕНИЕ** — «Последнее касание» для отчёта «без активности» = только записи `activities` (deadline/completed). `stage_history` и `deal.updated_at` касанием не считаем (иначе почти все сделки «свежие», отчёт при N=14 пуст).

**РЕШЕНИЕ** — Источники: при отсутствии строк `marketing_costs` по каналу ROMI не считаем делением на ноль — показываем «нет затрат» (referral/telegram/phone).

**РЕШЕНИЕ** — Порядок фаз изменён: живой контур (минимальный API + React + деплой за туннелем) поднимаем рано, сразу после core+transform; 7 отчётов делаем вертикальными срезами (SQL→API→страница) с инкрементальным деплоем; финал-доки и опц. Metabase — в конце. Причина: владелец хочет верифицировать данные и отчёты вживую по ходу, а не в самом конце.

**РЕШЕНИЕ** — Данные/инфраструктура: исходные CSV в git не коммитим (лежат локально в `data/`, всё живёт в БД); в README — инструкция «положи 13 CSV в `data/` → `make load`». Postgres на порту 5433. Все таблицы создаём средствами Python (SQLAlchemy + Alembic); `schema.sql` = дамп схемы. Metabase — в самый конец. Git-почта владельца остаётся как есть, историю не переписываем (почты в таблице `users` в git не попадут, т.к. данные не коммитим).

**ПРОВЕРКА** — Пересчитал дебиторку под новое правило корректировок: D1010=169000, D1005=−8000 (переплата), D1001=66000, D1002=84000 (pending 30000), D1003=0 — сходится с эталонами. Решения приняты до кода, по согласованию с владельцем.
