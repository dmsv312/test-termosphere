# AGENTS.md — правила разработки и документирования

Этот файл — рабочие правила для проекта (и для AI-агента, который помогает его строить). Проект ведётся в формате AI-first: разработчик ставит задачи, проверяет результат и принимает решения; AI генерирует код по спецификации. Цель правил — чтобы результат был воспроизводимым, проверяемым и честно задокументированным.

## Принцип работы

1. **Идём по шагам.** Одна задача — один понятный шаг (см. «План внедрения»). Не «сделай всё сразу».
2. **Каждый шаг реально проверяем.** После шага — запустить и свериться (количество строк, вывод команды, числа из отчётов сверять с эталонными расчётами). «Готово на словах» не принимается.
3. **Документируем по ходу, не постфактум.** Живой `docs/worklog.md` заполняется на каждом шаге. `AI_USAGE.md` собирается в конце из этого журнала.
4. **Не выдумываем данные.** Если значения нет — помечаем флагом, а не сочиняем. Грязные данные не удаляем молча — либо чиним, либо помечаем.

## Архитектура слоёв

- **raw** — таблицы `raw_*`, все колонки `TEXT`, **без ограничений** (PK суррогатный). Грузим CSV как есть. Причина: кривой формат даты/суммы не должен ронять импорт, и нужен оригинал для сверки.
- **core** — нормализованный слой: настоящие типы, `PK / FK / CHECK`, дефолты, справочники.
- Наши добавления к модели: справочник **`sources`** (нормализованные каналы) и таблица **`data_quality_issues`** (лог всех найденных проблем).

## Обработка качества данных — три реакции

Каждая найденная проблема получает одну из трёх реакций (это же поле `action` в `data_quality_issues`):

| Реакция | Когда | Что делаем |
|---|---|---|
| **fixed** (чиним) | есть однозначно правильное значение | нормализуем (канал, дата, телефон, ИНН, дефолт валюты) |
| **quarantined** (карантин) | данные бессмысленны (битая связь, дубль) | в core/отчёты не пускаем, но логируем (не удаляем) |
| **flagged** (флаг) | это бизнес-сигнал, а не грязь | оставляем и помечаем (отриц. сумма, WON без даты, неактивный менеджер) |

Полный каталог — в [DATA_QUALITY.md](DATA_QUALITY.md).

## Проверка связей — где и как

- **Активно — в Python-transform, ДО вставки в core.** Проверяем существование `deal_id / product_id / stage_id / company_id`, ловим дубли, чистим значения, принимаем решение fix/quarantine/flag и пишем в `data_quality_issues`.
- **Пассивно — ограничения БД в core (PK/FK/CHECK).** Это страховка: не дадут мусору попасть в core, даже если логика ошиблась. Но «умное» решение о том, что делать с плохой строкой, принимает Python.
- **raw ограничений не имеет** — принимает всё.

## Порядок загрузки (важно из-за FK)

Сначала справочники → потом сделки → потом дети сделок:

```
users, companies, contacts, products, pipeline_stages, sources
  → deals
    → deal_products, payments, activities, stage_history, production_orders, shipments
  → marketing_costs
```

## Стиль кода (backend)

- Деньги и ИНН — `Decimal` / `str`, не `float` (точность, ведущие нули).
- Функции-чистилки — маленькие, чистые, с юнит-тестами (нормализация канала/даты/телефона/суммы).
- Модели SQLAlchemy = целевая схема; миграции — через Alembic.
- Даты храним как `timestamptz` / `date`, часовой пояс данных — `+05:00`.

## Принятые решения (бизнес-логика)

Кратко (полностью и с обоснованием — в [ASSUMPTIONS.md](ASSUMPTIONS.md)):

1. **Сумма сделки** = `COALESCE(Σ позиций, expected_amount)`. Маржа — только по позициям.
2. **Дебиторка** = обычные оплаты `status='paid'`; `correction` уменьшает долг (остаток = сумма − оплачено + Σ correction); `pending` отдельной колонкой; LOST исключаем. Маржа/продажи по менеджерам — по WON (SHIPPED — отдельной пометкой).
3. **«Без активности»** — параметр `N=14`; дата среза = максимум по фактическим событиям = **2026-06-15** (без будущих плановых дат); касание = только `activities`. ROMI при 0 затрат = «нет затрат».
4. `discount` — абсолютная сумма в рублях. WON без `closed_at` — флаг (дату не выдумываем).

## План внедрения (по фазам)

Порядок пересмотрен: живой контур и деплой поднимаем рано, чтобы отчёты можно было верифицировать вживую по ходу.

| # | Шаг | Результат |
|---|---|---|
| 0 | Каркас | docker-compose (postgres:5435), Makefile, скелеты доков, инициализация worklog |
| 1 | raw-слой | raw-модели (TEXT) + загрузчик CSV → raw (CSV лежат локально в `data/`, в git не коммитим) |
| 2 | core-слой | core-модели (типы, PK/FK) + `sources` + `data_quality_issues`; миграция Alembic (таблицы — средствами Python) |
| 3 | transform | чистилки + проход raw → core (дедуп, связи, 3 реакции, лог) |
| 4 | Живой контур рано | минимальный FastAPI + React + docker-compose(api+web) + деплой за своим туннелем; первый экран — просмотр core + отчёт «проблемы данных» |
| 5 | Отчёты (вертикальными срезами) | каждый из 7 отчётов: SQL → API-эндпоинт → страница; инкрементальный деплой; сверка с эталонами; + `db/reports.sql`, `db/schema.sql` (дамп) |
| 6 | Финал-доки | README, DATA_QUALITY, ASSUMPTIONS, AI_USAGE, план дашборда, ERD, BPM/FSM |
| 7 | Опц. Metabase | BI поверх той же БД — только после основного объёма |

После каждого шага — запись в `docs/worklog.md`.

## Документирование

### Живой журнал `docs/worklog.md`

Append-only, датированные короткие записи с тегом:

- **РЕШЕНИЕ** — что выбрали, почему, какую альтернативу отклонили.
- **AI-ЗАДАЧА** — что отдали AI сделать.
- **ПРОВЕРКА** — что проверили вручную и чем (только реальные проверки).
- **ПРОБЛЕМА** — что нашли в данных / где AI мог ошибиться.

### `AI_USAGE.md`

Собирается в конце из `worklog.md`. Отвечает на: какие инструменты использовали, какие задачи отдавали AI, что проверяли вручную, где AI ошибался или мог ошибиться, какие решения приняли сами.

**AI-инструменты проекта:** Claude Code, ChatGPT Codex (OpenAI Codex).

Честность журнала обязательна: записи «ПРОВЕРКА» отражают только то, что реально запускалось и сверялось.
