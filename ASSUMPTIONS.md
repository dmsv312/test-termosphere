# ASSUMPTIONS.md — принятые допущения и решения

Здесь зафиксированы бизнес-решения и допущения, влияющие на отчёты и нормализацию. У большинства нет единственно правильного ответа — выбор сделан осознанно и объяснён.

## Отчётность

### 1. «Сумма сделки» — гибрид позиций и `expected_amount`

`amount = COALESCE(Σ позиций, expected_amount)`, где `Σ позиций = Σ(quantity · unit_price − discount)`.

- `expected_amount` — прогноз/оценка в карточке сделки; `deal_products` — фактический состав. Они **расходятся** (например, у одной сделки прогноз 150 000 при факте по позициям 126 000).
- У части сделок позиций нет вообще → для них берём `expected_amount` (иначе сумма занулилась бы).
- **Маржа считается только по позициям**: `Σ(quantity · unit_price − discount) − Σ(quantity · cost_price)`. Где позиций нет — «маржа неизвестна» + флаг.
- Дополнительно показываем расхождение план (`expected_amount`) / факт (позиции).

### 2. Дебиторка — учитываем только оплаченное

- «Оплачено» = `Σ` обычных оплат со статусом `paid` (prepayment/full). Кассовый принцип: деньги признаём при поступлении.
- Платежи типа `correction` — корректировка долга (возврат/пересчёт), а НЕ поступление денег: применяем к сумме к оплате. Итог: **остаток = `сумма − оплачено + Σ correction`** (correction со знаком). Пример: D1010 — единственная запись `correction −5000` при сумме 174000, обычных оплат нет → **остаток 169000** (не 179000). Прежнюю трактовку «минус уменьшает оплачено» отклонили: она давала долг больше суммы сделки — бессмыслица.
- В отчёте — отдельная колонка «корректировка» для прозрачности арифметики остатка.
- `pending`-оплаты в остаток **не включаем**, показываем отдельной колонкой «в ожидании».
- Остаток может быть отрицательным → **переплата** (флаг): напр. D1005 — оплачено 100000 при сумме 92000 → остаток −8000.
- Дебиторку считаем по обязующим стадиям (`contract / production / shipped / won`), стадию `lost` исключаем.

### 3. «Сделки без активности N дней»

- `N` — параметр отчёта, по умолчанию **14 дней**.
- «Сегодня» = **дата среза = максимум по фактическим событиям** (`updated_at / payment_date / actual_date / completed_at / changed_at`), без будущих плановых дат (`custom_deadline`, `planned_finish`). На этой выгрузке = **2026-06-15**. Не `CURRENT_DATE`: данные статичны. Та же дата среза используется в отчёте №4 (задержка производства).
- «Последнее касание» = `max(deadline_at / completed_at)` по записям `activities` сделки. `stage_history` и `deal.updated_at` касанием **не считаем** (иначе почти все сделки «свежие» и отчёт при N=14 пуст).
- Сделки вообще без активностей — включаем (последнее касание = «бесконечно давно»).
- Закрытые сделки (`won / lost`) исключаем.

## Нормализация

### 4. Скидка (`discount`) — абсолютная сумма в рублях

В данных ненулевая скидка встречается ровно один раз (5000 при позиции 20 × 1050 = 21000). Трактуем как рубли (процент дал бы бессмыслицу). **Защита:** если скидка превышает сумму позиции — флаг-аномалия.

### 5. Валюта по умолчанию — RUB

Все суммы в данных в рублях. Пустая `currency` → дефолт `RUB` + флаг.

### 6. Дедупликация сделок

При дубле `deal_id` оставляем строку с максимальным `updated_at`; вытесненную — в лог (`quarantined`).

### 7. Битые связи (сироты)

Сделки, которых нет в `deals`, но на которые ссылаются дети (позиции/оплаты/активности/производство), и ссылки на несуществующие товары/компании/стадии → **не выдумываем недостающую сущность**. Дети уходят в карантин (в отчёты не попадают) с логом; для полей вроде `company_id` возможен `NULL` + флаг.

### 8. WON без даты закрытия

Сделку не «чиним» выдумыванием даты — ставим флаг. (Возможная опция на будущее — вывести дату из перехода в WON в `stage_history`; по умолчанию не применяется.)

## Прочее

- Часовой пояс данных — `+05:00`; сохраняем как есть.
- Дубликат товара по SKU оставляем, но помечаем и связываем с каноническим (в отчётах по товарам не двоим).
- Неактивный менеджер, владеющий сделкой, — не ошибка данных, а бизнес-сигнал (флаг для управленческого отчёта).
- **Продажи/маржа по менеджерам** — считаем по выигранным (WON). SHIPPED-сделки (отгружено, но не закрыто — напр. D1005) показываем отдельной пометкой «отгружено, не закрыто»: не теряем фактическую продажу и не смешиваем с закрытыми.
- **Источники, ROMI при нулевых затратах** — если по каналу нет строк `marketing_costs` (referral/telegram/phone), ROMI не считаем делением на ноль: показываем «нет затрат».
